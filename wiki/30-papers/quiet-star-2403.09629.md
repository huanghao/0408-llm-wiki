# Quiet-STaR

**论文**：*Quiet-STaR: Language Models Can Teach Themselves to Think Before Speaking*（Zelikman et al., 2024）
**arxiv**：2403.09629
**机构**：Stanford University / Notbad AI Inc
**基座模型**：Mistral 7B

## 核心问题

**Chain-of-thought 推理很有效，但它只在有"问题-答案"对的时候才能用。能不能让模型在阅读任意文本时，也学会"在心里想一想再说"？**

**为什么 CoT 只能在 QA 对里用**：CoT 的训练信号来自"答案对不对"——模型生成一段推理链，最后给出答案，如果答案正确，就把这段推理链用于训练。这个机制需要两个前提：① 有一个明确的"问题"，② 有一个可以判断对错的"答案"。普通文本（新闻、论文、小说）既没有问题，也没有标准答案，所以传统 CoT 无法从这类文本里学推理。

举例对比：

| 文本类型 | 示例 | 能用传统 CoT 训练吗 |
|----------|------|-------------------|
| 数学题（QA 对） | "小明有 3 个苹果，吃了 1 个，还有几个？答案：2" | 能。答案"2"是明确的对错标准 |
| 维基百科文章 | "1969 年 7 月，阿姆斯特朗登上月球…" | 不能。没有"问题"，也没有"答案" |
| 代码注释 | "// 这里先排序再二分查找，避免重复遍历" | 不能。推理过程隐含在注释里，但没有对错标准 |

Quiet-STaR 的解法：不需要"答案对不对"，用"思考后是否能更好地预测下一个词"作为训练信号。维基百科文章里每个词的真实出现就是"答案"——如果在"阿姆斯特朗"之后生成一段"（他是第一个登月的人类，1969 年执行阿波罗 11 号任务）"的思考，能让模型更好地预测后续词"登上"，这段思考就值得被强化。

传统的推理训练方法（包括 STaR、CoT 微调）都依赖精心构建的问答数据集——模型在被问到问题时才触发推理。Quiet-STaR 的野心是：**让模型在处理每一个 token 时都悄悄思考**，用互联网普通文本（而不是题库）训练推理能力，让推理成为语言建模的一部分。

## 和 STaR 的关系

**STaR**（Zelikman et al., 2022，同一作者）是 Quiet-STaR 的前身：给定问答数据集，让模型生成 rationale，如果 rationale 导向了正确答案就保留并训练，迭代几轮后模型可以解更难的题。

**Quiet-STaR 的泛化**：去掉"需要问答数据集"这个前提。训练信号变成"这个 rationale 是否让模型更好地预测后续文本"——任何文本都可以用，不需要标注答案。

| | STaR | Quiet-STaR |
|--|------|-----------|
| 训练数据 | 问答数据集（需要标注） | 任意互联网文本 |
| 训练信号 | 答案是否正确 | 思考后是否能更好预测下一个词 |
| 推理触发 | 只在问题时触发 | 每个 token 后都触发 |
| 可扩展性 | 受限于数据集质量和覆盖 | 可从海量无标注文本中学习 |

## 方法：Think → Talk → Learn

Quiet-STaR 的训练循环分三步：

### Step 1：Think（并行生成思考）

在输入序列的**每个 token 位置**，模型插入一个 `<|startofthought|>` token，然后生成一段内部思考（rationale），最后插入 `<|endofthought|>` token。

**关键挑战**：如果对每个 token 分别做 forward pass，计算量是序列长度的平方，完全不可行。

**解法：tokenwise 并行采样**。利用 Transformer 的注意力掩码（attention mask）：构造一个对角掩码，使得每条"思考路径"只能看到自己之前的 token，不同位置的思考并行生成，每次 forward pass 同时推进所有位置的思考一个 token。相当于用一次 forward pass 同时延伸所有位置的思考，避免了顺序依赖。

**伪代码**（忽略并行掩码细节，展示核心逻辑）：

```python
for each token x_j in sequence:
    # Step 1: Think — 在 x_j 后插入思考
    thought = generate_tokens(
        context = [x_0,...,x_j, <startofthought>],
        length  = t  # 思考长度，超参数
    )
    thought += [<endofthought>]

    # Step 2: Talk — 两路预测 + 混合
    h_base    = hidden_state(x_j)                     # 不带思考的隐藏状态
    h_thought = hidden_state([x_j] + thought)         # 带思考后的隐藏状态

    p_base    = lm_head(h_base)                       # 不带思考的下一词分布
    p_thought = lm_head(h_thought)                    # 带思考的下一词分布
    w         = mixing_head(h_base, h_thought)        # 混合权重，0~1 标量

    p_talk    = w * p_base + (1-w) * p_thought        # 最终预测

    # Step 3: Learn — REINFORCE 奖励
    reward = log_prob(p_talk, true_next_tokens)       # 这条思考的得分
    #        - mean(log_prob across all thoughts)     # 减去平均基线
    if reward > 0:
        maximize log_prob(thought | context)          # 鼓励好的思考
```

实际实现中，所有位置的 Step 1 通过一个对角 attention mask 在单次 forward pass 里并行完成，不是逐个 token 串行跑。

**`<|startofthought|>` 的初始化**：嵌入向量初始化为破折号"—"的嵌入——因为破折号在文本中常表示"停顿/转折"，和"开始思考"的语义最接近。这两个 meta-token 是可学习的，它们的梯度权重设得比普通参数大（1e2 倍），让它们更快收敛。

### Step 2：Talk（混合预测）

思考完之后，模型对下一个 token 有两个预测：
- **不带思考的预测**：直接从原始 token 的隐藏状态预测（base prediction）
- **带思考的预测**：思考结束后从 `<|endofthought|>` 的隐藏状态预测

这两个预测通过一个**mixing head**（3 层 MLP）加权混合：

$$\log p^{\text{talk}} = w \cdot \log p^{\text{init}} + (1-w) \cdot \log p^{\text{thought}}$$

mixing head 的输入是"带思考的隐藏状态"和"不带思考的隐藏状态"拼接，输出一个 0–1 的标量权重 $w$。

**"思考"在模型里的本质**：思考不是额外的 Transformer 层，而是用**同一个模型**多生成一段文字。具体来说，模型读完 `x_j` 之后，先生成若干个"思考 token"（普通的文字 token，只是被 `<startofthought>`/`<endofthought>` 括起来），再把这段思考 token 和原始上下文一起送入模型，让模型在"已经看过这段内部思考"的状态下预测下一个词。

**和"不思考"的区别**：不思考时，模型的输入是 `[x_0, ..., x_j]`，预测 `x_{j+1}`；思考时，输入变成 `[x_0, ..., x_j, <startofthought>, t_1, ..., t_k, <endofthought>]`，用这个更长的上下文再预测 `x_{j+1}`。区别不在层数，而在**上下文里多了一段中间推导文字**，这段文字通过 Transformer 的 attention 机制影响最终的预测。本质上和"先写草稿再做题"是一回事——多出来的那段草稿文字帮助了最终判断。

**和"生成两段输出"的本质区别**：这是个很好的问题。表面上看，思考 token 确实是模型自己生成的普通 token，从序列角度和"输出两段内容"没有结构上的差别。区别在三点：

1. **目的不同**：思考 token 是"内部草稿"，不会出现在最终输出里，用户看不到。而两段输出都是给用户的。
2. **训练信号不同**：输出 token 用下一词预测损失（NLL）训练；思考 token 用 REINFORCE 训练——奖励是"这段思考有没有帮助后续预测"，而不是"这段文字本身对不对"。这意味着模型学到的是"什么样的思考有用"，而不是"思考要输出什么内容"。
3. **梯度流不同**：思考 token 的生成梯度来自 REINFORCE（间接信号），预测 token 的生成梯度来自 NLL 损失（直接信号）。两条路径优化的是不同的目标。

类比：让人先在草稿纸上算再填答案，和让人直接写两行答案，看起来都是"多写了一段"，但草稿不受"答案得对"约束，受的是"草稿有没有帮助你想清楚"的约束——这才让思考真正起到辅助推理的作用，而不是退化成另一段输出。

**为什么需要 mixing head**：训练初期，模型不知道怎么生成有用的思考，随机思考反而会干扰预测。mixing head 让模型可以在"思考没用时完全忽略思考（$w=1$）"和"思考有用时大量采纳（$w$ 趋近 0）"之间自适应切换。这避免了训练初期因为思考质量差导致整体性能崩溃。

### Step 3：Learn（REINFORCE 更新）

**REINFORCE 是什么**：REINFORCE（Williams, 1992）是强化学习里最基础的策略梯度算法，不是本论文提出的。核心思想极简：对于一个"不可微"的决策过程（比如生成离散 token），无法直接反向传播，但可以用"奖励信号 × 动作概率的梯度"来近似——如果某个动作（生成某段思考）得到正奖励，就提高它的生成概率；得到负奖励就降低。GRPO（DeepSeek-R1 用的算法）是 REINFORCE 的直接变体，见 [RLHF](../20-concepts/rlhf.md)。这里不单独写文档，因为 REINFORCE 本身一句话能说清楚，RLHF 文档里 PPO/GRPO 的部分已经覆盖了这套框架的完整背景。

训练信号来自"这次思考有没有帮助"——用 REINFORCE 算法：

$$r_j = \log p^{\text{talk}}_j(X_{j+1:j+n_{\text{true}}}) - \overline{\log p^{\text{talk}}_j}(X_{j+1:j+n_{\text{true}}})$$

奖励 = 这条思考下的对数似然 - 同位置所有思考的平均对数似然。如果这条思考比平均水平好（$r_j > 0$），就增加它的生成概率；反之不惩罚（只取正向奖励，训练更稳定）。

**非近视损失（non-myopic loss）**：奖励不只看下一个 token，而是看后续 $n_{\text{true}}$ 个 token 的联合对数似然——这让思考的价值可以体现在"帮助理解后面几步"上，而不只是猜对紧接着的那一个词。

举例：句子是"因此，根据勾股定理，斜边长度为 **5**"。如果只看下一个词，思考是否有帮助很难判断（下一词可能是常见词"5"或"根号"）；但如果看后续 4 个词的联合概率，一段写出"a=3, b=4, a²+b²=25"的思考就能获得明显更高的奖励。

**teacher-forcing 是什么**：在语言模型训练里，teacher-forcing 指在计算"预测第 k 步"时，把真实的第 k-1 步 token（而不是模型自己生成的 token）喂给模型。好处是避免误差累积——如果用模型自己预测的 token 递推，一旦第 1 步出错，后面每步都在错误基础上继续，梯度信号会很嘈杂。这里 Quiet-STaR 用 teacher-forcing 来计算"后续 $n_{\text{true}}$ 步的联合概率"：把真实的后续 token 依次插入上下文，每步都用真实值而非模型预测值，使得多步奖励信号稳定可信。

## 实验设置和结果

**训练数据**：
- 主实验：OpenWebMath（数学相关网页，推理 token 密度更高）
- 对照实验：C4（通用互联网文本）

**评估**：零样本直接回答，**不做任何下游任务微调**。

| Benchmark | 基座 Mistral 7B | Quiet-STaR（OpenWebMath） | Quiet-STaR（C4） |
|-----------|----------------|--------------------------|-----------------|
| GSM8K（零样本） | 5.9% | **10.9%** | 8.1% |
| CommonsenseQA（零样本） | 36.3% | **47.2%** | 42.6% |

**思考长度的影响**：性能随思考 token 数单调增加——更长的思考一致地带来更高准确率，说明 Quiet-STaR 确实在学习"如何思考"，而不只是在用占位 token 刷分。

**和 Pause Token 的对比**（Goyal et al., 2023）：
- Pause Token：在每个 token 后加一个不携带内容的"暂停 token"，CommonsenseQA 从 26.9% → 28.8%，GSM8K 反而下降
- Quiet-STaR：CommonsenseQA +10.9pp，GSM8K +5.0pp，且 token 越多效果越好

差距说明"真正生成语言内容的思考"比"形式上的暂停"有实质性价值。

**结合 Chain-of-Thought**：用 Quiet-STaR 模型做零样本 CoT（"Let's think step by step"），GSM8K 的 majority vote@8 从 40.6% → **47.7%**——内部静默思考和外部显式 CoT 是互补的，两者可以叠加。

**CoT 是不是只是 prompt 工程？**：是的，广义的 CoT（比如在 prompt 里加"Let's think step by step"）确实只是改变输入格式，不改变模型权重，属于 prompt 工程。但"CoT 微调"（在含有推理链的数据上 fine-tune 模型）会改变模型权重，不是纯 prompt 工程。Quiet-STaR 的思考是在**训练时**通过 REINFORCE 优化进模型权重的，既不是 prompt 工程（不需要在推理时写特殊 prompt），也不是人工标注推理链——而是模型从语言建模任务里自发学会的。这是它和普通 CoT prompting 最根本的区别。

## 思考对哪些 token 有帮助

Quiet-STaR 对大多数 token 几乎没帮助（在普通文本里，大部分 token 都很容易预测）。但对**难以预测的 token**，提升非常显著——呈现长尾分布：少数 token 获得了巨大的对数概率提升。

从生成的思考内容里可以看到，帮助最大的情形是：
- 需要回忆相关定理名称或公式的节点（如"下一个步骤用哪个定理"）
- 证明/推导里步骤转折的地方（"接下来应该做什么"）
- 需要理解语境才能判断下一词的歧义位置

## 局限性

1. **计算开销大**：每个 token 都生成思考，token 总数扩大了（思考长度 + 1）倍。Compute-adjusted 的比较（同等推理开销下）显示提升更小，见论文 Appendix C。

2. **只验证了 7B 规模**：Quiet-STaR 只在 Mistral 7B 上实验。论文承认，更大的模型可能有"收益不成比例地放大"的效果（历史经验如此），但未验证。

3. **不支持动态决定"何时思考"**：当前实现在每个 token 后都思考，不能跳过明显不需要思考的 token。如果 mixing head 能在**生成思考之前**预测"这里值不值得想"，就可以省掉大量无用思考——论文标注这是未来工作方向。

   **2026 年现状**：这个方向在两条路线上有进展——一是**模型内部解决**（推理模型如 o1/DeepSeek-R1 通过 RL 训练让模型自己学会在难题上多想、简单题少想，thinking budget 由模型根据题目动态分配，不是外部规则控制）；二是**推理时自适应**（test-time compute scaling，让模型在推理时多次采样或延长思考链，按需分配计算）。这两种方案都是"自然语言层面的动态思考"，而不是 Quiet-STaR 设想的"在 token 级别跳过思考"——后者目前没有主流落地。

4. **从零训练未验证**：所有实验都是在预训练模型上继续训练（continued pretraining）。从零开始用 Quiet-STaR 是否可行未知。

5. **思考的可解释性有限**：生成的思考是语言，看起来像推理，但不能保证它真实反映了模型的内部计算（faithfulness 问题）——这是所有 CoT 方法的共同局限。

## 现状与影响

**定性：奠基性工作，实现路线已被更简单的方案取代。**

Quiet-STaR 的核心思想——"推理能力可以从无标注普通文本中自发涌现"——在学术上被广泛引用，是 2024 年推理方向的重要参考点。但它的具体实现路线有一个工程上难以绕过的瓶颈：**每个 token 之后都插入思考**，计算量是正常推理的几十倍，在 Mistral 7B 上已接近极限，无法扩展到更大模型。

工业界随后选择了代价更低的路线——在**问题级别**生成一整段思考链（o1、DeepSeek-R1），用 RL 训练模型按难度分配思考长度。效果远超 Quiet-STaR 报告的 GSM8K 10.9%，工程复杂度也低得多。

**结论**：如果你在研究推理模型的训练方法，需要读这篇；如果你想复现或部署"让模型思考"的能力，不需要用 Quiet-STaR 的路线——直接看 DeepSeek-R1 或 o1 蒸馏方案更实用。

## 后续研究方向（2024–2025）

Quiet-STaR 提出的几个局限性，后续工作从不同角度有所推进：

**"思考"不必是自然语言**：Quiet-STaR 的思考是可读的文字 token。2024 年底的 Coconut（Chain of Continuous Thought，Hao et al.）走了更激进的路：把"思考"变成连续向量空间里的隐藏状态，直接把上一步的 hidden state 喂回模型作为下一步输入，完全跳过文字化。这让模型可以在连续空间里"同时保留多个可能的推理方向"（类似宽度优先搜索），而不是 CoT 那样线性地写出一条路径。代价是思考完全不可解释，但在需要大量搜索的逻辑推理任务上效率更高。

**推理模型对动态计算的解法**：o1、DeepSeek-R1 等推理模型用 RL 训练出了"按难度分配思考量"的能力——简单题直接答，难题自动展开更长的思考链。这是在自然语言层面解决动态计算，而非 Quiet-STaR 设想的 token 级跳过。两种思路的核心差异：Quiet-STaR 是"每个 token 都可以静默思考"，推理模型是"整道题统一分配一个思考预算"。

**计算开销问题**：Quiet-STaR 的思考在每个 token 之后都发生，计算量极大。实用系统（包括 o1 系列）的做法是只在问题级别触发一次长思考，而不是 token 级别——绕过了 Quiet-STaR 的并行采样难题，但也放弃了"用普通文本训练推理"这个核心野心。

## 和相关工作的关系

- **STaR（Zelikman et al., 2022）**：直接前身，Quiet-STaR 去掉了"需要问答数据集"的约束
- **Chain-of-Thought（Wei et al., 2022）**：Quiet-STaR 是"内化的 CoT"——思考不在输出里，是模型内部的静默过程；两者互补可叠加
- **[RLHF / GRPO](../20-concepts/rlhf.md)**：Quiet-STaR 用 REINFORCE 优化思考，但奖励信号是语言建模质量而非人类偏好；不需要额外奖励模型
- **[LIMO](./limo-2502.03387.md)**：LIMO 展示了精心挑选的 SFT 数据就能激发推理；Quiet-STaR 走了另一条路——用 RL 从普通文本里自发学习推理，两者是"数据效率 vs. 训练方法"上的不同取舍
