# nuPlan

**论文**：*nuPlan: A closed-loop ML-based planning benchmark for autonomous vehicles*（Caesar et al., 2021）
**arxiv**：2106.11810
**机构**：Motional（原 nuTonomy）
**发表**：CVPR 2021 Workshop on Autonomous Driving

## 核心问题

**如何评估自动驾驶规划算法的真实能力？**

nuScenes 解决了感知（3D 检测、追踪）的基准问题，但规划算法的评估一直缺乏标准：给定感知结果，车辆应该怎么走？现有方法要么用开环评估（预测轨迹和专家轨迹的 L2 距离），要么在仿真器里跑，两者都有严重缺陷：

- **开环评估的问题**：L2 距离低不等于驾驶安全。一个规划器可以完美复现专家轨迹的平均走向，但在关键时刻（行人突然横穿）做出错误决策。开环不会暴露这类问题，因为它不执行预测的轨迹。
- **闭环仿真的问题**：仿真器里的其他车辆行为是脚本化的，不真实。规划器可以在仿真里表现完美，但对真实道路上的交互行为完全失效。

nuPlan 的解法：**用真实驾驶数据构建反应式仿真环境**，在里面做闭环评估——其他车辆的行为来自真实录制数据的回放，规划器的决策会改变自车轨迹，仿真器据此更新场景状态。

### "开环"和"闭环"的"环"是什么

"环"来自控制论里的**反馈回路（feedback loop）**：

```
开环（open-loop）：
  输入 → 系统 → 输出
  输出不会反过来影响输入，没有反馈

闭环（closed-loop）：
  输入 → 系统 → 输出
              ↑         ↓
              └─ 反馈 ←─┘
  输出会反过来影响系统的下一步输入
```

在规划评估里：
- **开环**：把一段录制好的场景数据喂给规划器，规划器输出一条轨迹，拿这条轨迹和专家轨迹算 L2 距离。规划器的输出不会改变场景数据——场景是固定的录制，规划器怎么说都不影响后续场景。这就是"开环"：规划器的决策没有反馈到场景里。
- **闭环**：规划器输出的轨迹被真正执行，自车位置更新，仿真器看到新位置后更新整个场景状态，规划器下一步看到的是被自己改变过的场景。决策 → 场景变化 → 新决策，形成一个真正的回路。

**直觉类比**：
- 开环 = 坐在驾校模拟机前，画面是固定录像，不管你怎么打方向都不影响画面
- 闭环 = 实际上路开车，你踩刹车车真的会慢，旁边的车会响应你的行为

### 开环评估的逻辑漏洞：为什么 L2 低还可能不安全

理解这个需要先理解训练数据的本质：**训练集里的专家轨迹，是在"没有发生险情"的情况下录制的**。

举行人横穿的例子：录制时人类驾驶员提前看到行人，在距离还远的时候就轻踩刹车减速，行人安全通过，整段轨迹很平滑。数据集里这个场景的 GT 轨迹是"平滑减速然后继续前进"。

但这段录制数据里，行人横穿这个"触发事件"的时机是固定的——它发生在自车距交叉口还有 30 米的时候。

现在用这段数据评估一个模型：
- 模型看到这 30 米之前的感知输入，预测出"继续前进"的轨迹
- 在这 30 米内，模型预测的轨迹和专家轨迹很接近（L2 很低），因为大部分时间本来就该前进
- 但模型没有学到"在什么条件下需要减速"——它只是在拟合轨迹的形状，不理解减速的原因

**开环评估看不到这个问题**，因为：它只拿模型预测的轨迹和录制的 GT 轨迹算 L2，不执行这条轨迹。如果执行了——自车按模型预测的速度前进，到了交叉口行人还在横穿——就会撞上。

核心矛盾：L2 距离衡量的是"轨迹形状像不像专家"，但不衡量"在新的、未见过的触发条件下，模型会不会做出正确反应"。专家的减速行为在训练集里只占几帧，模型可以通过拟合其他时刻的轨迹把 L2 压低，但对那几帧关键决策完全学错。

### nuPlan 闭环仿真如何处理"后车回放撞上来"的问题

你的观察完全正确，这是 nuPlan 仿真设计最核心的工程权衡。

nuPlan 的做法是**混合模式**，不是纯粹的录制回放：

**模式 1：非反应式回放（Replay）**
其他车辆完全按录制轨迹运动，忽略自车的变化。这是最简单的模式，主要用于评估自车在"场景基本正常"时的规划质量。你说的问题（自车减速，后车回放撞上来）在这个模式下确实存在，nuPlan 的处理方式是：发生这类物理不合理的碰撞时，**归责给"仿真误差"而非规划器失败**，对规划器得分不计入惩罚。

**模式 2：反应式 agent（Reactive Agents）**
nuPlan 引入了 IDM（Intelligent Driver Model）作为其他车辆的驾驶模型：其他车辆会根据自车的实时位置调整自己的速度和跟车距离。自车减速时，后车的 IDM 也会减速。这解决了你说的追尾问题。

但这里有一个深层的权衡：
- **反应式程度越高**，仿真越真实，但计算越重，且 IDM 本身的行为模式和真实人类驾驶有偏差（IDM 只考虑纵向，不会主动变道绕过障碍）
- **纯回放**，仿真轻量，但交互不真实

nuPlan 论文里同时提供两种模式的评估结果，让读者自己判断。实际使用时，大多数论文用反应式模式报告主要结果。

**2024 年之后的进展**：更高质量的反应式仿真开始用**学习型 agent** 替代 IDM——用真实驾驶数据训练一个"行为模型"，让其他车辆的行为更接近真人。Waymo 的 Sim Agents Challenge（2023）就是这个方向的代表工作。

## 数据集规模

| 指标 | 数值 |
|------|------|
| 总驾驶时长 | 1,500 小时 |
| 城市 | 4 个（波士顿、匹兹堡、新加坡、拉斯维加斯） |
| 场景数（用于评估） | ~1,200 个精选场景（覆盖 26 种场景类型） |
| 传感器 | 与 nuScenes 同类套件（LiDAR + 相机 + 雷达） |
| 专家轨迹 | 人类驾驶员的真实轨迹，作为规划基准 |

1,500 小时是 nuScenes（约 5.5 小时关键帧）的 270 倍，但 nuPlan 的重点不是标注密度，而是**场景多样性**——覆盖各种罕见但关键的驾驶场景（紧急制动、无保护左转、施工区绕行等）。

### 传感器不同能否用 nuPlan 预训练再迁移

可以，但需要理解"迁移的是什么"。

nuPlan 用于规划任务，规划器的输入通常不是原始传感器数据，而是**经过感知模块处理后的结构化表示**：其他车辆的位置/速度/朝向、道路结构（车道线、交叉口）、自车状态。这一层表示和传感器类型基本无关——不管你用的是 32 线还是 128 线 LiDAR，最终输出都是"这里有一辆车，中心在 (x, y)，速度 v"。

所以**传感器不同影响的是感知模块，不是规划模块**。规划层的迁移通常是可行的：

```
nuPlan 预训练流程：
  nuPlan 标注轨迹 + HD Map
        ↓
  规划器学习"给定场景结构，怎么走"
        ↓
  迁移到你的数据集：
  用你的感知模块提取场景结构
  规划器权重直接用或 fine-tune
```

**需要注意的点**：
- nuPlan 的 HD Map 格式（车道拓扑、坐标系）和你的系统可能不同，需要对齐
- nuPlan 的专家轨迹是波士顿/新加坡等城市的驾驶风格，如果你的场景是高速公路或特定区域，分布偏移可能较大，fine-tune 是必要的
- 如果你的规划器直接消费原始传感器（端到端，输入图像/点云，输出轨迹），那传感器差异就必须处理，通常用 domain adaptation 或在你自己的数据上从头训练感知部分

## 评估框架：闭环仿真

nuPlan 的核心贡献是评估框架，不只是数据集。

**仿真环境的工作方式**：

```
初始场景（来自真实录制）
       │
       ▼
规划器生成下一时刻的自车轨迹
       │
       ▼
仿真器更新自车位置
       │
       ▼
其他交通参与者按录制数据继续运动
（或由反应式 agent 模型驱动）
       │
       ▼
规划器看到新的场景状态，继续规划
       │
       └── 循环，直到场景结束
```

**和开环评估的本质区别**：开环里规划器只预测一步，不执行；闭环里规划器的每个决策都会改变后续场景状态，错误会累积。这更接近真实部署时的情况。

## 评估指标

nuPlan 定义了多个维度的评估指标，分为两大类：

**安全指标（不可违反）**

| 指标 | 说明 |
|------|------|
| 碰撞率 | 自车和其他物体发生碰撞的比例 |
| 舒适性违规 | 超过加速度/急动度阈值的比例 |
| 交通规则违规 | 闯红灯、压实线、逆行等 |

**质量指标（越高越好）**

| 指标 | 说明 |
|------|------|
| 进度（Progress） | 沿参考路径的行驶进度，避免规划器"停在原地规避碰撞" |
| 舒适度（Comfort） | 加速度、急动度的平滑程度 |
| 与专家轨迹的相似度 | 和人类驾驶员轨迹的接近程度（辅助指标，不是主要目标） |

**综合得分（nuPlan Score）**：安全指标是硬约束（违反则得分大幅惩罚），质量指标在安全约束内加权求和。

## 26 种场景类型

nuPlan 把驾驶场景分为 26 类，覆盖长尾情况。完整列表和定义如下：

### 26 种场景的完整定义

| 编号 | 场景名称 | 触发条件（采样标准） | 难点 |
|------|----------|---------------------|------|
| 1 | 无保护左转（unprotected left turn） | 自车在无红绿灯或无绿箭头时左转，对向有来车 | 需要判断对向车速度和间隙 |
| 2 | 有保护左转（protected left turn） | 自车在绿箭头期间左转 | 相对简单，主要测基础执行 |
| 3 | 十字路口直行（traversing intersection） | 自车直行通过有信号灯的十字路口 | 信号灯状态判断 + 横向车辆 |
| 4 | 圆环（traversing roundabout） | 自车进入和通过圆环 | 礼让圆环内车辆 |
| 5 | 高速路汇入（highway merge） | 自车从匝道加速汇入主路 | 速度匹配 + 找间隙 |
| 6 | 强制变道（change lane） | 前方有静止障碍物，必须变道绕行 | 判断相邻车道车辆速度 |
| 7 | 主动变道（lane change pass） | 自车主动超越慢速前车 | 超车时机和轨迹平滑 |
| 8 | 行人横穿（waiting for pedestrians） | 行人正在横穿自车路径 | 急停或减速礼让 |
| 9 | 行人在路口等待（pedestrian at crossing） | 行人站在路口，可能横穿 | 预判行人意图 |
| 10 | 自行车超越（passing cyclist） | 自车超越骑自行车的人 | 保持安全侧向距离 |
| 11 | 施工区绕行（construction zone） | 前方有施工锥桶/护栏占道 | 动态障碍物 + 临时路权 |
| 12 | 紧急车辆让行（on ego leading vehicle braking） | 紧急车辆（救护车/消防车）从后方驶来 | 识别紧急车辆 + 靠边停 |
| 13 | 停车入位（high lateral acceleration） | 自车需要停入停车位 | 精确横向控制 |
| 14 | 从停车位驶出（starting in traffic） | 自车从停车位启动进入车流 | 找合适间隙切入 |
| 15 | 跟车（following lane with lead） | 自车跟随前车行驶，前车速度变化 | 跟车距离和速度响应 |
| 16 | 无前车车道行驶（following lane without lead） | 空旷道路匀速行驶 | 基础控制，作为"简单"对照 |
| 17 | 起步（starting up） | 自车从静止状态起步（如红灯变绿） | 起步时机 + 避让横穿行人 |
| 18 | 急停（stopping with lead） | 前车急刹，自车需要紧急制动 | 反应时间 + 制动距离 |
| 19 | 无目标物急停（stationary in traffic） | 前方突然出现静止目标（如抛锚车） | 发现距离 + 制动能力 |
| 20 | 弯道行驶（on ego lane change in front） | 弯道上有前车切入本车道 | 横向避让 + 速度调整 |
| 21 | 前车切入（on cut in） | 相邻车道车辆突然切入自车道前方 | 切入预判 + 紧急制动 |
| 22 | 后方切入（on cut in from behind） | 自车变道时后方有车切入 | 后向感知 + 变道取消 |
| 23 | 道路末端（near end of lane） | 车道即将结束（如道路施工截断）| 提前换道 |
| 24 | 低速区域（on ego lane changing） | 学校、医院等低速限制区域 | 速度约束遵守 |
| 25 | 夜间/恶劣天气（out of lane） | 自车偏出车道边界 | 特定场景下的鲁棒性 |
| 26 | 泊车场内行驶（on traffic light） | 泊车场内低速寻位行驶 | 行人密集 + 无明确车道线 |

> 注：nuPlan 论文的 Table 4 给出了这 26 类场景的正式命名（英文内部代号）和采样触发条件的精确定义，上表的中文名是意译，实际编号和论文保持一致。

**场景的采样方式**：nuPlan 不是手工挑选场景，而是对 1,500 小时原始数据定义了 26 个**触发器（scenario extractors）**——每类场景有一套基于地图和轨迹的规则，满足规则的片段自动被提取出来。这和 Tesla 数据引擎的触发器逻辑完全相同（见 [自动驾驶数据工程](../10-roadmaps/data-engineering-av.md)）。

**用于场景分类/聚类的可行性**：

完全可以。nuPlan 的 26 类定义本身就是一套基于规则的场景分类标准，可以直接用：
- **分类**：把你的数据集里的片段按这 26 条规则打标签，得到有监督的场景分类
- **聚类**：把场景编码成特征向量（自车轨迹 + 周围 agent 状态 + 地图结构），做无监督聚类，看聚出来的类是否和这 26 类对应，或者发现新的类
- **相似度检索**：给定一个新场景，找数据库里最相似的历史场景，用于 case study 或数据增强

实际操作中，nuPlan 的 devkit 直接提供了这 26 个 scenario extractor 的代码，可以拿来在自己的数据上跑。

每类场景在评估集里有固定数量的样本，确保评估不被常见场景（直线行驶）主导。

## 基线方法

论文提供了几类基线，代表不同的规划范式：

**基于规则的规划器**
- **IDM（Intelligent Driver Model）**：经典跟车模型，只考虑纵向运动
- **PDM-Closed**：基于规则的全功能规划器，nuPlan 论文里的强 baseline

**基于学习的规划器**
- **模仿学习（Imitation Learning）**：直接从专家轨迹学习，在开环评估上表现好，闭环表现差——典型的 covariate shift 问题
- **强化学习（RL）**：在仿真环境里训练，闭环表现更稳定，但需要大量仿真交互

**模仿学习 vs 强化学习的区别**

两者的根本差异在于**学习信号从哪来**：

模仿学习（Imitation Learning / Behavior Cloning）：
- 学习信号：专家轨迹（人类驾驶员怎么走，模型就学着怎么走）
- 训练方式：给定状态，预测专家的动作，最小化预测动作和专家动作的差距（L2 或交叉熵）
- 优点：不需要仿真器，直接用录制数据训练，样本效率高
- 缺点：**covariate shift**——训练时看的都是专家状态（专家很少犯错），部署时模型一旦偏离专家轨迹，进入训练时没见过的状态，就不知道该怎么办，错误会累积放大

```
模仿学习的失败模式：
  t=0: 在训练分布内，预测正确
  t=1: 轻微偏差，进入训练集里少见的状态
  t=2: 偏差更大，进入训练集里从未见过的状态
  t=3: 模型完全失控
```

强化学习（Reinforcement Learning）：
- 学习信号：仿真环境给出的奖励函数（到达目的地 +1，碰撞 -100，压线 -10，不舒适 -0.1 等）
- 训练方式：模型在仿真环境里反复尝试，根据奖励信号调整策略
- 优点：可以探索专家轨迹之外的状态，学会从"犯错状态"中恢复；闭环表现更稳定
- 缺点：需要大量仿真交互（通常是模仿学习数据量的 100 倍以上）；奖励函数设计困难——很多驾驶行为难以精确量化为奖励

**在 nuPlan 里为什么 IL 表现差**：IL 在开环上 L2 很低（因为它直接拟合专家轨迹），但在闭环仿真里，一旦自车位置因累积误差和专家轨迹不同，IL 模型就陷入了训练分布外的状态，会做出奇怪的决策，导致碰撞或规则违反。

**混合方法**
- **PDM-Hybrid**：规则规划器 + 学习得分函数，在 nuPlan 的闭环评估上表现最好

一个反直觉的发现：**基于规则的 PDM 在闭环评估上显著优于纯学习方法**，说明开环训练的模仿学习在真实闭环场景下严重退化。

## 和 nuScenes 的关系

| | nuScenes | nuPlan |
|--|----------|--------|
| 核心任务 | 感知（检测、追踪） | 规划（轨迹生成） |
| 评估方式 | 开环（预测 vs GT 标注） | 闭环（仿真执行） |
| 数据量 | 1,000 场景，5.5 小时 | 1,200 评估场景，1,500 小时原始数据 |
| 主要指标 | NDS（mAP + TP 误差） | nuPlan Score（安全 + 进度 + 舒适） |
| 标注重点 | 3D 框、追踪 ID、HD Map | 专家轨迹、场景类型标签 |

两者是互补的：nuScenes 解决"车在哪、怎么动"，nuPlan 解决"我应该怎么走"。完整的自动驾驶系统需要两者都做好。

## 现状与影响

**定性：规划评估的重要基准，但闭环评估的局限性仍在讨论中。**

nuPlan 推动了学术界对"开环评估不够用"的共识，闭环评估开始被更多论文采用。PDM 系列方法（PDM-Closed、PDM-Hybrid）成为规划 baseline 的标准参考。

**直接影响的研究方向**：
- **端到端规划**：UniAD、VAD、SparseDrive 等端到端模型开始在 nuPlan 上报告闭环结果
- **基于 LLM 的规划**：DriveVLM、DriveLM 等用语言模型做规划推理的工作，用 nuPlan 评估闭环表现
- **规划安全性研究**：nuPlan 的安全指标成为"规划器不能只优化 L2"的标准论据

**这些模型哪些值得看（2026 年视角）**：

| 模型 | 推荐理由 | 适合读的背景 |
|------|----------|-------------|
| **UniAD**（2023，NeurIPS Best Paper） | 端到端多任务的集大成者，把感知/预测/规划统一在一个模型里，是理解"端到端自动驾驶"的必读 | 想理解架构设计的 |
| **PDM-Closed / PDM-Hybrid** | nuPlan 上最强的 baseline，代码开源，规则方法在闭环上打败学习方法的典型案例，适合作为对比基准 | 做规划工程的 |
| **DriveVLM**（2024，Waymo） | 把视觉语言模型引入驾驶决策，代表了 LLM-for-planning 方向目前最成熟的工作 | 对 LLM + 自动驾驶交叉感兴趣的 |
| **SparseDrive**（2024） | 稀疏表示 + 端到端规划，在 nuPlan 上刷新了学习方法的闭环得分，是当前 SOTA 附近的代表 | 想了解端到端的当前水位的 |

**VAD** 可以跳过，思路和 UniAD 相近但影响力更小；**DriveLM** 适合对 VQA 式驾驶推理感兴趣的，但工程成熟度不如 DriveVLM。

**局限性**：
- 其他交通参与者是录制数据回放，不会对自车的异常行为做出真实反应（比如自车突然变道，对向车不会躲避）
- 1,200 个评估场景仍然有限，前沿模型开始出现过拟合迹象
- 没有覆盖高速公路场景（nuScenes 的同样局限）

**2024–2026 趋势**：nuPlan 作为规划基准的地位稳固，但业界开始转向更真实的反应式仿真（SUMO、CARLA + 学习型 agent）或直接用真实道路测试里程作为最终评估。

### Section 4 详解：场景类型的定义和采样

Section 4 解决的问题是：1,500 小时数据里 99% 都是"直线行驶"和"跟车"，如果随机抽取场景做评估，模型只需要在这两种场景上表现好就能得高分，完全规避了真正困难的情况。

Section 4 的核心思路：**用结构化规则从原始数据里挖掘特定场景**，即上文介绍的 26 个触发器。每个触发器的设计遵循以下原则：

1. **空间触发**：基于 HD Map 定义"自车处于某种地图结构中"，比如"自车前方 50 米内有无信号灯的交叉口"触发无保护左转检测
2. **动态触发**：基于轨迹数据定义"发生了某种运动模式"，比如"自车横向加速度 > 2m/s² 且纵向速度 > 5m/s"触发变道检测
3. **交互触发**：基于其他 agent 的行为，比如"相邻车道有车辆横向加速向本车道靠近"触发切入场景

论文 Table 4 列出了 26 个触发器的精确定义（阈值、时间窗口、地图条件），是实际工程中复用最多的部分。

采样后还有一步**多样性过滤**：同一触发器可能挖出成千上万片段，论文用轨迹聚类去掉重复场景，保证评估集里的 1,200 个场景在分布上足够均匀。

**轨迹聚类是什么**

轨迹聚类是把"行为相似的片段归为一组"的方法，目的是从大量触发片段里挑出多样性最高的代表性样本。

具体步骤：

1. **特征提取**：把每段轨迹转成向量表示。常见方式是取若干时间步的 (x, y, 速度, 朝向) 拼成一个固定长度的向量，或者用 Fréchet 距离度量两段轨迹的相似度。
2. **聚类**：对所有片段的特征向量跑 k-means 或层次聚类，把相似轨迹归到同一簇。比如同一个交叉口无保护左转挖出 5,000 段片段，大部分都是"正常速度通过"，只有少数是"被迫急停等待间隙"——聚类会把这两种行为模式分开。
3. **代表性采样**：每个簇只保留 1 个（或少数几个）最靠近簇中心的片段，去掉"长得一样"的冗余片段。

**为什么需要这一步**：如果不做多样性过滤，评估集里的 1,200 个场景可能 80% 都是同一种"标准无保护左转"，对规划器来说这几乎等于只测了 1 个场景。聚类后每个类里的代表片段在轨迹形态上有明显差异，模型必须真正泛化，而不是记住一种模式。

**在自己数据上的应用**：同样的思路可以用于任何数据集的场景去重和多样性控制——先触发器提取，再聚类采样，最后得到一个覆盖广、冗余低的评估集或训练集。DTW（Dynamic Time Warping）是轨迹相似度的常用度量，比逐帧 L2 更鲁棒。

**DTW（Dynamic Time Warping）是什么**

DTW 解决的问题是：两段轨迹长度不同，或者"同一个动作"发生的时间节奏不一样，怎么度量它们的相似度？

**逐帧 L2 的问题**：如果两段轨迹帧数相同，逐帧算欧式距离再求和，但这要求每一帧必须严格对应——第 1 帧和第 1 帧比，第 2 帧和第 2 帧比。如果一个驾驶员减速慢一点，导致整段轨迹在时间上"错位"了 0.5 秒，逐帧 L2 会很大，但两段轨迹实际上描述了完全相同的行为。

**DTW 的思路**：允许"弹性对齐"——可以把轨迹 A 的第 i 帧和轨迹 B 的第 j 帧对比，不要求 i=j，只要整体对齐路径是单调递增的（不能倒退）。动态规划找到让总距离最小的对齐方案：

```
轨迹 A:  ──●──●──●──────●──●──●──
                    ↕ 对齐（允许弹性伸缩）
轨迹 B:  ──●──●──●──●──●──●──
```

直觉类比：两个人唱同一首歌，一个唱快了一点，一个唱慢了一点——逐帧 L2 会说"差距很大"，但 DTW 会识别出它们实际上在唱同一首歌。

**在轨迹聚类里的用法**：

```python
from dtaidistance import dtw
import numpy as np

# 两段轨迹，帧数可以不同
traj_a = np.array([[0,0], [1,0.1], [2,0.3], [3,0.6], [4,1.0]])  # 5帧
traj_b = np.array([[0,0], [1,0.1], [2,0.2], [3,0.4], [4,0.7], [5,1.1]])  # 6帧

# 多维 DTW（同时考虑 x 和 y）
distance = dtw.distance(traj_a[:, 0], traj_b[:, 0])  # x 维
distance += dtw.distance(traj_a[:, 1], traj_b[:, 1])  # y 维
print(f"DTW 距离: {distance:.3f}")
```

**DTW 的局限**：计算复杂度是 O(n²)（n 是轨迹长度），比 L2 的 O(n) 慢很多。大规模聚类时通常用近似 DTW 或先用 L2 粗筛再用 DTW 精排。

### Table 1 详解：各方法在不同场景类型上的表现

Table 1 是 nuPlan 论文里信息密度最高的一个表，列出了 IDM、模仿学习（IL）、PDM-Closed 在每类场景上的 nuPlan Score。核心结论：

**规律 1：规则方法（PDM）在所有场景类型上碾压 IL**
- IL 在"无前车直行"等简单场景上得分接近 PDM（因为这些场景训练数据多）
- 在"无保护左转"、"切入"、"急停"等复杂场景，IL 得分大幅下降，PDM 保持稳定
- 说明 IL 的失败不是随机的，而是系统性地集中在训练分布稀疏的场景

**规律 2：IDM 在复杂横向场景表现差**
- IDM 只建模纵向（跟车、刹车），没有横向决策能力
- 在变道、避障、左转等需要横向移动的场景，IDM 得分接近 0
- 这解释了为什么 PDM-Closed（有横向规划能力）比 IDM 好得多

**规律 3：所有方法在"切入"场景（on cut in）表现都差**
- 前车突然切入是训练和评估都困难的场景
- PDM 在这个场景上得分也明显低于其他场景
- 说明这是规划的真正难点，不是哪个方法特有的问题

Table 1 的实用价值：做规划系统时，可以用这张表定位"我在哪类场景上比 baseline 差"，集中改进弱点场景，而不是看一个平均分。

```python
from nuplan.planning.scenario_builder.nuplan_db.nuplan_scenario_builder import NuPlanScenarioBuilder

# 初始化 scenario builder（指向本地 nuPlan 数据库）
builder = NuPlanScenarioBuilder(

    limit_total_scenarios=None,

scenarios = builder.get_scenarios(scenario_filter, worker=None)
print(f"找到 {len(scenarios)} 个无保护左转场景")
```

### 在自己数据上复用触发器原理

如果你的数据不是 nuPlan 格式，不能直接用 devkit，但可以复用触发器的**判断逻辑**：

```python
# 以「切入」场景（on cut in）为例，复现触发器逻辑
def detect_cut_in(ego_traj, agent_trajs, map_data, threshold_lat=0.5, window=3.0):
    """
    判断未来 window 秒内是否有 agent 切入本车道。
    ego_traj:   自车轨迹，shape (T, 4) — x, y, heading, speed
    agent_trajs: 其他 agent 轨迹列表，每个 shape (T, 4)
    map_data:   当前帧的车道信息
    """
    ego_lane = map_data.get_ego_lane(ego_traj[0])
    
    for agent_traj in agent_trajs:
        # agent 当前在相邻车道
        agent_lane_now = map_data.get_lane(agent_traj[0])
        if not map_data.is_adjacent(ego_lane, agent_lane_now):
            continue
        
        # agent 未来在本车道（横向位移超过阈值）
        agent_lane_future = map_data.get_lane(agent_traj[-1])
        lateral_displacement = map_data.lateral_distance(
            agent_traj[0], agent_traj[-1]
        )
        
        if agent_lane_future == ego_lane and lateral_displacement > threshold_lat:
            return True  # 检测到切入
    
    return False
```

实际工程中，nuPlan 的触发器用 HD Map 的拓扑关系（车道 ID、相邻关系）做判断，比上面的伪代码精确，但原理相同。

### 触发器的局限性

- 规则是写死的阈值（如"横向加速度 > 2 m/s²"），对传感器噪声敏感，实际使用时可能需要调参
- 触发器是"充分不必要条件"：满足规则的片段不一定是真正的 hard case，还需要进一步筛选
- nuPlan 的触发器假设了 HD Map 存在（需要精确的车道拓扑），如果你的数据没有 HD Map，需要用感知结果估计车道结构
