# DataComp-LM (DCLM)

**论文**：*DataComp for Language Models*（Li et al., 2024）
**arxiv**：2406.11794
**机构**：多机构合作（UW、Apple、CMU 等）

## 核心问题

数据质量对 LLM 性能的影响被严重低估，但缺乏统一的对照实验基准来客观评估不同数据策略的效果。现有模型（Llama、Mistral 等）大多不公开训练数据细节，导致无法区分"数据更好"和"模型/算力更好"。

DCLM 的目标：**固定模型架构和训练配方，只改变数据，系统比较不同数据处理策略的效果**。

## 主要贡献

**DCLM-POOL**：240T token 的原始网页语料，基于 Common Crawl，用 resiliparse 做 HTML 提取（比 trafilatura 快 8 倍）。

**DCLM-BASELINE**：在 DCLM-POOL 上应用一套过滤 pipeline，得到的高质量子集：

1. **启发式清理**：URL 过滤、语言检测、页面长度过滤，清除约 20% 的原始文档
2. **去重**：MinHash 去重（小规模）+ 改进的 Bloom filter（超大规模，更易扩展）

   **MinHash vs Bloom filter 的区别**：两者解决的问题不同。MinHash 用来找"近重复"——两篇文章内容相似但不完全一样（改写、转载），通过比较 Jaccard 相似度来判断；Bloom filter 用来找"精确重复"——某条 URL 或某段文本是否已经见过，是一个"见过/没见过"的集合查询结构。Bloom filter 的特点是极省内存（用概率换空间，有小概率误报但不会漏报），适合在超大规模下快速过滤已见过的内容。两者互补：先用 Bloom filter 过滤精确重复，再用 MinHash 处理近重复。
3. **模型质量过滤**：[fastText](../20-concepts/fasttext.md) 分类器（OH-2.5 + ELI5），只保留得分最高的 10% 文档

   **OH-2.5 和 ELI5 是什么**：这是两个 fastText 分类器的训练数据来源，不是两种不同的算法。
   - **OH-2.5**（OpenHermes 2.5）：一个高质量指令微调数据集，包含大量经过筛选的问答和对话文本，用作"高质量文本"的正例
   - **ELI5**（Explain Like I'm 5）：Reddit 上的一个子版块，专门用简单语言解释复杂概念，文本质量高、信息密度大，也用作正例

   两个数据集的正例混合在一起训练同一个 fastText 分类器，分类器学到的是"像这类高质量文本"的特征，然后给 DCLM-POOL 里的每篇文档打分，只保留 top 10%。之所以叫"fastText 分类器"，是因为模型结构就是 fastText——分类任务，只是训练数据不同。

**竞赛框架**：5 个规模档（400M 到 7B 参数），用统一的 OpenLM 训练框架，53 项 benchmark 评估，让研究者在相同条件下比较数据策略。

## 关键结果

| 模型 | MMLU 5-shot | 训练 token |
|------|-------------|------------|
| DCLM-BASELINE 7B | 64% | 2.6T |
| Llama 3 8B | 66% | — |
| MAP-Neo（前 SOTA） | ~57% | — |

- 相比 MAP-Neo 提升 6.6 个百分点
- 与 Llama 3 8B 性能相当，但计算量少 6.6 倍
- 在 7B 规模下，数据过滤策略的选择可造成 35–44% 的精度差异

**小规模可预测大规模**：10 种方法在不同规模间的 Pearson 相关系数 0.83–0.95，说明用小模型做数据策略实验是可靠的。

## 关键技术细节

**最优质量过滤方案**：fastText 分类器，训练数据为 Wikipedia + OpenWebText2 + RedPajama Books 混合，保留得分 top 10% 的文档。这比 PageRank、SemDedup、AskLLM 等方案都更好。

这里被比较的几个方案都是"用什么信号来判断文档质量"的不同思路：
- **PageRank**：用网页的入链数量作为质量信号——被很多其他页面引用的页面，质量可能更高。这是搜索引擎的经典方法，但对于 LLM 训练数据未必准确（高 PageRank 的页面不一定文字质量好）
- **SemDedup**：用语义嵌入向量做去重，把语义相似的文档聚类后只保留一个，目标是提高数据多样性而不是直接过滤低质量文档
- **AskLLM**：用一个语言模型（通常是较小的 LLM）直接判断文档质量，给每篇文档打分。准确但计算代价高，在 TB 级数据上不现实

**污染防控**：检测训练数据与 MMLU、HellaSwag 等 benchmark 的文本重合，移除匹配页面。

## 和 ccNet / Llama 3 的关系

- 和 ccNet 思路一致：都强调质量过滤，都用 fastText 分类器
- DCLM 的贡献是**系统对比**而不是提出新方法——用 416 个基线实验证明数据策略是性能瓶颈

  **"系统对比"是怎么系统的**：在 DCLM 之前，每篇论文都在自己的设置下测自己的数据方法，无法横向比较——A 论文说"我的过滤方法让模型好了 3%"，B 论文说"我的方法好了 5%"，但两者用的模型大小、训练 token 数、评估 benchmark 都不同，根本不知道谁真的更好。

  DCLM 的做法是固定所有变量，只改数据：
  - 固定模型架构（同一套 Transformer）
  - 固定训练 token 数（每个规模档固定）
  - 固定 benchmark（53 项评估）
  - 固定超参数（同一套 OpenLM 配置）

  在这个框架下跑了 416 个实验，覆盖了各种数据策略组合（不同的去重方法、不同的质量过滤方法、不同的数据来源混合比例）。因为所有变量都控制了，不同实验的结果才可以直接比较，得出"方法 A 比方法 B 好 X%"这样有意义的结论。
- Llama 3 的数据处理和 DCLM-BASELINE 的思路高度重叠

## 现状与影响

**定性：仍活跃，是数据配方研究的重要基准框架。**

DCLM（2024 年 6 月）提出的核心贡献——固定模型、只改数据、系统对比——已成为后续数据研究的参考范式。DCLM-BASELINE 本身也被多个开源项目采用作为起点数据集。

fastText + 高质量正例（OH-2.5/ELI5）的质量过滤方案，是目前在算力和效果之间最均衡的开源方案之一，仍在实际使用中。

**局限**：DCLM 的结论基于英文网页文本，多语言场景的泛化性未充分验证。竞赛框架的设置（固定架构、固定超参）也意味着结论对特定训练配置有依赖。
