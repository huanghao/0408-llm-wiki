# 域专属管道：Code 和 Math

## 是什么

为代码和数学内容建立独立于通用网页管道的数据处理流程。核心思路是：代码和数学文本的结构、token 分布、质量标准都与自然语言有根本性差异，用通用管道处理会系统性地损害这类数据的质量。

## 为什么需要独立管道

**通用管道的三个问题**：

1. **HTML 解析破坏结构**：通用解析器优化的是新闻、博客等文章类内容，对代码块的缩进、数学公式的符号处理很差。比如 `<pre>` 标签里的代码可能被当作普通文本处理，导致换行和空格丢失。

2. **质量分类器偏差**：基于"Wikipedia 引用"或"书籍相似度"训练的质量分类器，会把代码文件打低分——因为代码不像"正常文本"。这会导致高质量代码数据被过滤掉。

3. **去重粒度不匹配**：代码的重复模式和自然语言不同。GitHub 上大量仓库包含相同的 boilerplate（`README`、`LICENSE`、`setup.py`），但同一个函数在不同仓库里的实现可能是独立有价值的。文档级去重对代码不够精细。

## Llama 3 的做法

参考 DeepSeek-AI et al.（2024）的方案，使用在 Llama 2 标注数据上 prompt-tuned 的 DistilRoBERTa 分类器，专门识别：
- 包含数学推导的页面
- STEM 领域的推理内容
- 代码与自然语言交织的内容（如技术教程）

同时对这类内容定制 HTML 提取规则和启发式过滤，而不是走通用管道。

## 2026 年的进展：从过滤到生成

今天的代码/数学管道已经不只是"从互联网过滤高质量内容"，而是主动生成数据：

**代码**：
- 用强模型生成函数、补全代码、编写测试用例
- 用代码执行器实际运行生成的代码，验证输出是否正确（execution-based filtering）
- 只保留能通过测试的代码样本

**数学**：
- 用强模型生成解题过程（chain-of-thought）
- 用形式化验证工具（如 Lean、Isabelle）验证证明步骤的正确性
- 或用多个模型交叉验证答案，取多数票

这个方向的关键洞察是：**互联网上高质量的数学推理过程本来就很少**，过滤再好也有天花板；而生成 + 验证可以突破这个天花板，只要 verifier 可靠。

## 为什么这个思路越来越重要

随着 reasoning 能力成为模型核心竞争力，代码和数学数据的质量直接决定模型上限。DeepSeek-R1、Qwen 系列等在推理上表现突出的模型，都在代码/数学数据上投入了大量工程。

这也解释了为什么"域专属管道"不是一个可以后来补上的优化，而是从一开始就需要设计进去的核心组件。
