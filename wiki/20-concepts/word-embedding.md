# Word Embedding（词嵌入）

## 是什么

Word embedding 是把词（或任意离散特征）映射成稠密向量的技术。

```
"猫"  → [0.2, -0.5, 0.8, 0.1, ...]   # 100 维向量
"狗"  → [0.3, -0.4, 0.7, 0.2, ...]   # 和"猫"很近
"汽车" → [-0.6, 0.9, -0.2, 0.8, ...]  # 和"猫"很远
```

训练完成后，语义相近的词对应的向量也相近（可以用余弦相似度衡量）。

## 为什么需要它

机器学习模型不能直接处理文字，只能处理数字。最朴素的做法是 one-hot 编码：词表里有 10 万个词，就用一个 10 万维的向量，对应词的位置填 1，其余填 0。

one-hot 的问题：
1. **维度灾难**：10 万维向量极其稀疏，计算效率低
2. **没有语义信息**："猫"和"狗"的 one-hot 向量距离和"猫"与"汽车"完全一样，模型无法利用语义相似性

Word embedding 用低维稠密向量（通常 50–300 维）解决这两个问题。

## 核心思想：分布假说

> 一个词的含义由它的上下文决定。（Distributional Hypothesis, Firth 1957）

"猫"和"狗"经常出现在相似的上下文里（"我养了一只___"，"___在睡觉"），所以它们的向量应该相近。这个直觉是所有 word embedding 方法的基础。

## 训练过程

以 word2vec 的 Skip-gram 模型为例（最经典的方法）：

**任务设定**：给定中心词，预测它周围的词（上下文窗口内）。

```
句子：  "我  喜欢  吃  苹果  和  香蕉"
窗口=2：以"吃"为中心，预测 ["喜欢", "苹果"]（左右各2个词）
```

**训练步骤**：

1. 初始化嵌入矩阵（随机数）
2. 取中心词"吃"的向量，预测上下文词的概率
3. 真实上下文词（"苹果"）的概率应高，其他词应低
4. 计算损失，反向传播，微调"吃"和"苹果"对应的向量行
5. 在整个语料上重复数百万次

训练结束后，嵌入矩阵里每一行都是对应词的向量表示。

**负采样加速**：

正常训练时，每次更新需要对全词表（比如 10 万个词）计算 softmax，即"这个词出现在这里的概率 / 所有词出现在这里的概率之和"——分母要遍历 10 万个词，代价极高。

负采样的做法：不算全词表，只随机抽 5 个"不应该出现在这里的词"作为反例。每步变成 6 个二分类问题：

```
正例：(center="吃", context="苹果")  → 标签=1，这对词确实共现，sigmoid 应输出接近 1
负例：(center="吃", context="汽车")  → 标签=0，随机采样，sigmoid 应输出接近 0
负例：(center="吃", context="电脑")  → 标签=0
... （共 5 个负例）
```

标签完全自动生成：正例从文本里的真实词对取，负例按词频随机采样，不需要人工标注。计算量从 O(10万) 降到 O(6)，速度提升万倍量级。

## 手搓实现

Skip-gram 就是 word2vec 的一种模式（另一种是 CBOW，见下文）。完整代码见 `tools/word_embedding_scratch.py`。

**X 和 Y 是什么**：

```
句子："我 喜欢 吃 苹果 和 香蕉"，窗口=2

以"吃"（ID=2）为中心词，生成训练样本：
  X（中心词 ID）= 2
  Y_正例（上下文词 ID）= [1("喜欢"), 3("苹果")]   ← 真实共现的词
  Y_负例（随机采样 ID）= [5("汽车"), 8("电脑"), ...]  ← 随机采的，标签=0

把正例和负例拼在一起：
  center_ids:  [2]          shape (B,)
  context_ids: [1, 5, 8, ...]  shape (B, 1+N_NEG)，第0列是正例
  labels:      [1, 0, 0, ...]  第0列=1，其余=0
```

**模型结构**：

```python
class SkipGram(nn.Module):
    def __init__(self, vocab_size, dim):
        super().__init__()
        self.W_in  = nn.Embedding(vocab_size, dim)  # 中心词嵌入矩阵
        self.W_out = nn.Embedding(vocab_size, dim)  # 上下文词嵌入矩阵

    def forward(self, center_ids, context_ids):
        # center_ids:  (B,)              中心词 ID
        # context_ids: (B, 1+N_NEG)     第0列正例，后N_NEG列负例
        center_vecs  = self.W_in(center_ids)    # (B, DIM)        查表
        context_vecs = self.W_out(context_ids)  # (B, 1+N_NEG, DIM) 查表
        # 每个中心词向量和对应的上下文向量做内积
        # 内积越大 → sigmoid 越接近 1 → 预测"这两个词共现"
        return torch.einsum('bd,bnd->bn', center_vecs, context_vecs)  # (B, 1+N_NEG)
```

**Loss：BCEWithLogitsLoss**：

```python
labels = torch.zeros(B, 1 + N_NEG)
labels[:, 0] = 1.0   # 第0列（正例）标签=1，其余（负例）标签=0

loss = nn.BCEWithLogitsLoss()(logits, labels)
```

`BCEWithLogitsLoss` = Binary Cross Entropy with Logits，二元交叉熵损失。把每个（中心词, 上下文词）对视为一个二分类问题：这两个词是真实共现（1）还是随机凑的（0）？

内部计算：`loss = -[y * log(σ(z)) + (1-y) * log(1-σ(z))]`，其中 z 是内积，σ 是 sigmoid。
- 正例（y=1）：内积越大，loss 越小 → 模型被推着让真实共现的词向量靠近
- 负例（y=0）：内积越小，loss 越小 → 模型被推着让随机词对的向量远离

这两个力的合力，让语义相近的词（常共现）向量靠近，语义无关的词向量远离。

**为什么有两个嵌入矩阵 W_in 和 W_out**：中心词和上下文词扮演不同角色，用两套参数效果更好。训练完后通常只用 W_in 作为最终词向量。

运行后可以看到：
- 训练过程中 loss 如何下降（约 0.96 秒完成 50 epoch）
- 训练完成后语义相近的词向量余弦相似度更高（汽车↔自行车 0.86，猫↔苹果 0.19）
- 任意特征（不只是词）都可以用同样方式训练嵌入

## Skip-gram vs CBOW vs word2vec

这三个概念的关系：

- **word2vec** 是 2013 年 Google 提出的框架，包含两种训练模式
- **Skip-gram**：给定中心词，预测上下文词。对低频词效果更好，训练慢一些
- **CBOW**（Continuous Bag of Words）：给定上下文词，预测中心词。训练更快，对高频词效果更好

```
Skip-gram：  中心词 → 预测 → 上下文词
CBOW：       上下文词 → 预测 → 中心词（方向相反）
```

手搓代码实现的是 Skip-gram。CBOW 的区别只是把 X 和 Y 互换：输入是多个上下文词的平均向量，输出是中心词。

## 推广：不只是词

Word embedding 的思路可以推广到任何离散特征：

| 特征 | 应用场景 | 怎么定义"共现"或"相似" |
|------|----------|------------------------|
| 词（word） | NLP，文本分类 | 在同一句话的窗口内出现 |
| 字符 n-gram | fastText subword | 属于同一个词 |
| 用户 ID | 推荐系统（user embedding） | 购买/点击了相同商品 |
| 商品 ID | 推荐系统（item embedding） | 被同一用户购买/点击 |
| 节点 ID | 图神经网络（node embedding） | 在图上随机游走路径中相邻 |
| 类别特征 | 结构化数据建模 | 和目标变量（标签）的关联 |
| 场景 ID | 自动驾驶场景检索 | 感知输出相似、行为相似（见 [场景相似性](./scene-embedding.md)） |

核心逻辑完全相同：把离散 ID 映射成稠密向量，通过"共现"或"相似行为"定义相似性，用梯度下降训练。

## Embedding Lookup 的本质

**"查表"等价于 one-hot 乘矩阵**：

给定词 ID=2，嵌入矩阵 A 的 shape 是 `(V, DIM)`，"查第 2 行"等价于用 one-hot 向量左乘矩阵：

```
one-hot (1×V)  ×  A (V×DIM)  =  结果 (1×DIM)

[0, 0, 1, 0, 0]  ×  [[-0.1,  0.2,  0.5],   =  [0.3, -0.1, 0.8]
                      [ 0.4, -0.3,  0.1],
                      [ 0.3, -0.1,  0.8],   ← 只有第2行（系数=1）被选中
                      [-0.2,  0.6, -0.4],
                      [ 0.1,  0.0,  0.2]]
```

两者数学等价，但直接取行（`A[2]`）不需要任何乘加运算，只是一次内存寻址，速度极快。

**稀疏更新**：

训练时，反向传播的梯度只流向这次 forward 用到的行。一条文本涉及几十个特征 ID，就只更新几十行，其余几十万行梯度为 0、不更新。这叫稀疏更新，让 embedding 的训练代价和输入长度成正比，而不是和词表大小成正比。

## 在高性能场景的使用

当你需要对海量数据做特征提取（TB 级，实时推理），embedding lookup 是最快的特征计算方式：

- **时间复杂度**：O(1)——给定 ID，直接取矩阵对应行，不做任何计算
- **可以预计算**：把所有可能的特征 ID 对应的向量提前算好存起来
- **稀疏更新**：训练时每步只更新用到的少数行，大词表也能高速训练

fastText 之所以快，核心就是用 embedding lookup 代替了复杂的特征计算。如果你有一批离散特征需要高速处理，可以考虑同样的思路：先离线训练 embedding，推理时只做查表。

## 局限性

1. **静态向量**：每个词只有一个向量，无法处理一词多义（"苹果"是水果还是公司）
2. **词表外的词（OOV）**：训练时没见过的词没有向量（fastText 用字符 n-gram 缓解这个问题）
3. **需要大量语料**：低频词的向量质量差，因为上下文样本少

BERT 等模型用上下文动态生成向量，解决了问题 1，但速度慢得多。
