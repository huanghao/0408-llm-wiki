# RLHF（Reinforcement Learning from Human Feedback）

## 是什么

RLHF 是一种训练方法，让语言模型从人类的偏好反馈中学习，使模型从"预测下一个词"转变为"给出人类认为有用、无害、诚实的回答"。

**为什么需要它**：预训练模型（GPT-3 等）只学会了"写出统计上合理的文字"，但不会"按人的意图回答问题"。给它一个问题，它可能给出一段流畅但完全没用的续写。RLHF 解决的是这个**对齐（alignment）**问题——让模型的行为和人类意图对齐。

这个需求直接来自 Gopher 等论文的发现：规模扩大能提升能力，但不能解决有害输出和偏见问题。

## 核心流程

RLHF 分三个阶段：

### 阶段一：监督微调（SFT）

收集人工编写的高质量问答对，用这些数据对预训练模型做监督微调，让模型初步学会"如何回答问题"的格式和风格。

```
数据：(问题, 人工写的好答案) 对
训练：标准的交叉熵损失，和预训练一样
目的：让模型有个合理的起点，后续 RL 才能有效优化
```

**数据量**：InstructGPT 用了约 **13,000 条**人工标注的问答对做 SFT。这个数量看起来很少（预训练用了几千亿 token），但 SFT 的目的不是让模型学新知识，而是调整输出格式和风格——预训练已经学会了知识，SFT 只是告诉模型"用这种方式表达"。后续研究（如 LIMO）发现几百条高质量数据就能有效激发推理能力，说明 SFT 的数据质量比数量更重要。

### 阶段二：训练奖励模型（Reward Model）

让模型对同一个问题生成多个答案，人工标注员对这些答案进行排序（哪个更好）。用这些偏好数据训练一个奖励模型（RM），让它能自动给模型输出打分。

```
数据：(问题, 答案A, 答案B, 人工标注"A比B好")
训练：Bradley-Terry 模型，优化 P(A > B) = sigmoid(RM(A) - RM(B))
目的：把人类偏好压缩成一个可自动计算的分数函数
```

奖励模型通常是一个和语言模型架构相同但在最后加了一个打分头的模型（把最后一层的输出从"词表大小"换成"1 个标量分数"）。

**数据量**：InstructGPT 用了约 **33,000 条**偏好比较对训练奖励模型（每条是"问题 + 两个答案 + 哪个更好"）。RM 的数据量通常比 SFT 多，因为偏好标注比写答案更快（标注员只需要比较，不需要从头写），可以更高效地收集。

### 阶段三：PPO 强化学习

用奖励模型作为环境，用 PPO（Proximal Policy Optimization）算法优化语言模型，让它生成能获得高奖励分数的输出。

```
策略（policy）：语言模型本身（每步选择下一个 token）
奖励（reward）：奖励模型给完整回答的打分
约束：KL 散度惩罚，防止语言模型偏离 SFT 版本太远（避免 reward hacking）
```

**KL 惩罚为什么重要**：如果没有约束，模型会找到"让奖励模型打高分但实际质量很差"的输出——比如生成一堆重复的高分词。KL 惩罚让模型不能偏离 SFT 版本太远，保持语言质量。

## InstructGPT：RLHF 的经典实现

**论文**：*Training language models to follow instructions with human feedback*（Ouyang et al., 2022，OpenAI）

InstructGPT 是第一个系统性应用 RLHF 的大语言模型，也是 ChatGPT 的直接前身。

关键发现：
- 1.3B 参数的 InstructGPT 在人类评估中优于 175B 的 GPT-3——**参数少 100 倍，但对齐更好**
- 说明对齐训练的效果不亚于规模扩大
- 人类标注员认为 InstructGPT 更诚实、更无害、更有用

## Constitutional AI（CAI）

**论文**：*Constitutional AI: Harmlessness from AI Feedback*（Bai et al., 2022，Anthropic）

**动机**：RLHF 依赖大量人工标注，成本高，且人工标注本身可能不一致。Constitutional AI 的想法是：**用 AI 替代部分人工标注**。

**做法**：

1. 给模型一套"宪法"——一组原则，如"不要提供有害信息"、"要诚实"、"尊重用户自主性"
2. 让模型自己生成回答，然后根据宪法原则评判自己的回答，并修改
3. 用模型的自我评判结果（AI 反馈）训练奖励模型，替代部分人工偏好标注
4. 再用这个奖励模型做 RLHF

**和 RLHF 的区别**：

| | RLHF | Constitutional AI |
|--|------|-------------------|
| 偏好来源 | 人工标注员排序 | AI 根据宪法原则打分 |
| 成本 | 高（需要大量人工） | 低（AI 自动生成反馈） |
| 可解释性 | 低（人工偏好是黑盒） | 高（宪法原则明确可查） |
| 效果 | 经过验证，效果好 | 和 RLHF 相当，更可扩展 |

Constitutional AI 是 Anthropic Claude 系列模型的核心训练方法。

## RLHF 的局限性

1. **Reward Hacking**：模型可能找到让奖励模型打高分但实际质量差的输出，KL 惩罚只能部分缓解
2. **标注一致性**：不同标注员对"好答案"的判断不一致，奖励模型学到的是平均偏好，不一定是最优偏好
3. **奖励模型过拟合**：奖励模型本身可能对某些模式过拟合，语言模型会过度迎合这些模式
4. **计算成本高**：PPO 训练需要同时在内存里维护多个模型（策略模型、参考模型、奖励模型、价值模型），显存压力大

## DPO（Direct Preference Optimization）

**论文**：Rafailov et al., 2023（arxiv: 2305.18290）

**核心思想**：RLHF 需要先训练奖励模型，再用 PPO 优化语言模型，流程复杂、计算成本高。DPO 的发现是：**可以直接从偏好数据推导出语言模型的训练目标，跳过奖励模型这一步**。

**数学等价性**：在 RLHF 中，最优策略（语言模型）和奖励模型之间有一个解析关系：

$$r(x, y) = \beta \log \frac{\pi^*(y|x)}{\pi_{\text{ref}}(y|x)} + \beta \log Z(x)$$

DPO 把这个关系代入偏好损失函数，消掉了奖励模型，得到一个直接用偏好对优化语言模型的损失：

$$L_{DPO} = -\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right)$$

其中 $y_w$ 是"好答案"，$y_l$ 是"差答案"，$\pi_{\text{ref}}$ 是 SFT 模型（参考策略）。

**直觉**：损失函数鼓励模型给好答案更高的对数概率（相对于参考策略），给差答案更低的对数概率。不需要奖励模型，直接用偏好对训练。

**和 RLHF 的对比**：

| | RLHF + PPO | DPO |
|--|-----------|-----|
| 奖励模型 | 需要单独训练 | 不需要 |
| 训练复杂度 | 高（4 个模型同时在内存，见下） | 低（2 个模型：策略 + 参考） |
| 稳定性 | PPO 超参数敏感 | 稳定，像普通监督训练 |
| 数学关系 | 近似 | 精确等价 |
| 实践效果 | 略好（更灵活） | 相当，更易复现 |

DPO 目前是学术界和中小团队的主流选择；大公司（OpenAI、Google）有资源做 PPO，效果可能略好。

**PPO 的 4 个模型是哪些**：

PPO 训练时需要同时在显存里维护：
1. **策略模型（policy）**：正在被训练的语言模型，生成回答
2. **参考模型（reference）**：SFT 模型的冻结副本，用于计算 KL 散度惩罚，防止策略偏离太远
3. **奖励模型（reward model）**：给生成的回答打分
4. **价值模型（value model / critic）**：估计每一步的期望回报，用于计算优势函数

这 4 个模型通常都和策略模型一样大（至少奖励模型和价值模型是同等规模），所以训练一个 70B 的模型，显存里实际需要装下 4 × 70B 的参数，对硬件要求极高。DPO 只需要策略模型和参考模型（2 个），GRPO 需要策略 + 参考 + 奖励（3 个，去掉了价值模型）。

## GRPO（Group Relative Policy Optimization）

**来源**：DeepSeek-R1 论文（2025），用于训练推理模型。

**动机**：PPO 需要一个**价值模型（value model/critic）**来估计每个状态的期望回报，用于计算优势函数（advantage）。价值模型和策略模型一样大，训练时需要同时维护，显存压力极大。GRPO 的目标是**去掉价值模型**。

**核心思想**：对同一个问题生成一组答案（group），用这组答案的奖励均值作为基线，每个答案的优势 = 它的奖励 - 组内均值。不需要价值模型估计期望回报，直接用组内相对排名。

$$A_i = \frac{r_i - \text{mean}(r_1, \ldots, r_G)}{\text{std}(r_1, \ldots, r_G)}$$

**和 PPO 的对比**：

| | PPO | GRPO |
|--|-----|------|
| 价值模型 | 需要（和策略模型一样大） | 不需要 |
| 优势估计 | 价值模型预测期望回报 | 组内相对奖励 |
| 显存占用 | 高（策略 + 参考 + 奖励 + 价值，4 个模型） | 低（策略 + 参考 + 奖励，3 个模型） |
| 适用场景 | 通用 | 有明确奖励信号的任务（数学、代码） |

**为什么适合推理任务**：数学和代码有明确的对错——答案对了奖励 1，错了奖励 0。这种稀疏但明确的奖励信号让组内相对比较非常有效：同一道题生成 8 个答案，4 个对 4 个错，对的优势为正，错的优势为负，梯度方向清晰。对于开放式对话任务，奖励信号不那么明确，GRPO 效果可能不如 PPO。

DeepSeek-R1 用 GRPO 训练出了具有 chain-of-thought 推理能力的模型，证明了纯 RL（不需要蒸馏）也能激发推理能力。

## 附录：PPO 的梯度机制和监督学习的区别

**问题**：PPO 的训练和普通神经网络的反向传播（梯度下降）是一个体系吗？

**答案**：用的是同一套反向传播工具，但目标函数的来源完全不同。

### 监督学习的梯度

普通监督训练（如 SFT）的目标是最小化交叉熵损失：

$$L_{SFT} = -\sum_t \log P_\theta(y_t \mid x, y_{<t})$$

每个 token 都有一个确定的"正确答案"（标注数据里的 token），损失函数直接对比预测和真实值。梯度通过反向传播从最后一层传回到每个参数。

### PPO 的梯度

PPO 的目标是最大化期望奖励，但奖励是在**完整回答生成完之后**才给出的（奖励模型打分），不是逐 token 的。这带来一个问题：**奖励信号是稀疏的**——模型生成了几百个 token，但只在最后得到一个分数，不知道哪个 token 对最终分数贡献最大。

PPO 的做法：

1. **用价值模型估计每步的期望回报**：价值模型 $V(s_t)$ 预测"从当前状态出发，未来能得到多少奖励"，把稀疏的最终奖励"分摊"到每一步
2. **计算优势函数**：$A_t = r_t + \gamma V(s_{t+1}) - V(s_t)$，衡量"这一步的实际回报比预期好多少"
3. **Clip 约束**：PPO 的目标函数对策略更新幅度做了限制，防止一次更新太大导致训练不稳定

$$L_{PPO} = -E_t\left[\min\left(\frac{\pi_\theta(a_t|s_t)}{\pi_{old}(a_t|s_t)} A_t,\ \text{clip}\left(\frac{\pi_\theta}{\pi_{old}}, 1-\epsilon, 1+\epsilon\right) A_t\right)\right]$$

4. **反向传播**：这个损失函数对 $\theta$（语言模型参数）求梯度，用普通的 Adam 优化器更新——**和监督学习用的是同一套反向传播机制**

### 关键区别

| | 监督学习（SFT） | PPO |
|--|---------------|-----|
| 梯度工具 | 反向传播 | 反向传播（相同） |
| 损失来源 | 每个 token 有标注 | 完整回答后才有奖励 |
| 信号密度 | 密集（每 token 一个梯度） | 稀疏（最终奖励分摊到每步） |
| 需要额外模型 | 不需要 | 需要价值模型估计每步回报 |
| 训练稳定性 | 稳定 | 超参数敏感，容易不稳定 |

**直觉**：监督学习是"告诉模型每步应该输出什么"；PPO 是"告诉模型最终结果好不好，让它自己摸索每步该怎么做"。工具相同（反向传播），但信号的来源和密度完全不同。DPO 的优势就在于把 RL 的稀疏奖励问题转化成了监督学习的密集信号形式。

### PPO 和监督学习的核心区别

PPO 用的是同一套反向传播工具，但信号来源完全不同：

- **监督学习**：每个 token 都有确定的"正确答案"，损失函数逐 token 计算，梯度密集
- **PPO**：奖励在完整回答生成后才给出，信号稀疏——模型不知道哪个 token 对最终得分贡献最大

PPO 用**价值模型**解决这个问题：价值模型估计"从当前步出发，未来能得到多少奖励"，把稀疏的最终奖励分摊到每一步，再用**优势函数**（实际回报 - 预期回报）指导每步的梯度方向。

**Clip 约束**防止单次更新幅度过大：每次更新时，新策略和旧策略的概率比被限制在 $[1-\epsilon, 1+\epsilon]$ 范围内，超出范围的梯度被截断。

可运行 demo 见 `tools/ppo_demo.py`（极简 PyTorch 实现，5 步序列生成任务，75 epoch 后 avg_reward 从 1.25 收敛到接近 5）。概念详解见 [PPO Demo 逐行讲解](./ppo-explained.md)。
