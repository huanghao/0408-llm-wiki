# 合成数据 + 验证

## 是什么

一种数据生成范式：用强模型生成大量候选训练数据，再用 verifier（验证器）过滤出正确的部分，而不是依赖从互联网爬取和过滤真实数据。

## 为什么出现

传统管道（包括 Llama 3）的逻辑是从互联网爬取数据，然后过滤出高质量子集。这个逻辑有一个结构性天花板：**互联网上高质量的数学推理过程、正确的代码、严谨的科学推导，本来就不多**。

具体来说：
- 互联网上的数学内容，大多数是题目和最终答案，完整的推理过程很少
- 高质量代码（有测试、有文档、逻辑正确）在 GitHub 上是少数
- 过滤再好，也只能从有限的高质量真实数据里选，无法凭空创造

合成数据 + 验证打破了这个天花板：生成的数量可以任意多，质量由 verifier 保证。

## 核心机制

**为什么生成比验证便宜**：强模型生成 100 个候选答案，其中可能只有 10–20 个是正确的。但用 verifier 把这 10–20 个筛出来，成本远低于让人工标注 10–20 个正确答案。

**verifier 的类型**：

| 类型 | 适用场景 | 可靠性 |
|------|----------|--------|
| 代码执行器 | 代码生成 | 高（执行结果是客观的） |
| 形式化验证（Lean/Isabelle） | 数学证明 | 高（但覆盖范围有限） |
| 答案匹配 | 有标准答案的数学题 | 中（答案对不代表过程对） |
| Process Reward Model（PRM） | 推理链逐步验证 | 中（依赖 PRM 质量） |
| 多模型交叉验证 | 通用推理 | 低（多数模型同时出错的情况存在） |

## 代表性工作

**DeepSeek-R1（2025）**：大规模使用 RL + verifier 生成高质量推理数据，在数学和代码 benchmark 上达到 GPT-4 级别。是这条路线最有影响力的成果，让"合成数据 + RL"进入主流视野。

**Numina Math（2024）**：专门为数学竞赛题（AMC、AIME、奥数）生成解题过程，用答案匹配和形式化验证过滤，是这类任务上效果最好的训练数据之一。

**WizardCoder / Evol-Instruct（2023）**：用强模型对已有代码问题做"进化"（增加复杂度、添加约束），生成更难的训练样本，再用执行验证过滤。

**Self-Play Fine-Tuning（SPIN，2024）**：用模型自己生成的数据和真实数据做对比学习，不需要外部 verifier，但效果有限。

## 局限

**只在有明确正确答案的领域有效**：数学和代码有客观的对错标准，verifier 可以可靠地工作。但对开放性写作、常识推理、价值判断等任务，很难定义"正确"，verifier 的可靠性大幅下降。

**生成数据的分布依赖生成模型**：如果生成模型在某类问题上有盲点或偏见，生成的数据会继承这些问题，导致训练出来的模型在这些问题上同样有缺陷。

**规模化成本仍然很高**：调用强模型生成大量数据，API 成本或计算成本很高，对资源有限的团队不友好。

**answer-level verification 不等于 process-level verification**：答案正确不代表推理过程正确。用答案匹配做 verifier 会保留"答案碰巧对但推理有误"的样本，对模型推理能力的提升有限。Process Reward Model（PRM）可以缓解这个问题，但 PRM 本身的训练也需要大量标注。

## 和传统管道的关系

合成数据 + 验证不是替代传统管道，而是补充：

- 通用语言能力（流畅性、知识广度）：仍然主要依赖大规模真实数据
- 推理、代码、数学能力：合成数据 + 验证已经成为主要来源
- 指令跟随、对话能力：合成数据（GPT-4 蒸馏）+ 人工偏好标注
