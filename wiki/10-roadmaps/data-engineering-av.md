# 自动驾驶数据工程

自动驾驶的数据挑战和 LLM 有一个根本性的相似——**数据飞轮**：更好的模型发现更多 corner case → 标注 → 训练更好的模型。但具体挑战不同：标注成本高 3 个数量级、长尾场景稀缺、多模态时间戳对齐。

## 必看：理解数据引擎（Data Engine）范式

1. **Tesla AI Day 2021 / Karpathy CVPR 2021**（演讲，非论文）  
   搜索："Andrej Karpathy CVPR 2021 Tesla FSD"  
   看点：业界最成熟的数据引擎描述——从部署车队自动挖掘 hard case，到人工标注，到模型改进的完整飞轮。这个范式和 Llama 3 用小模型实验预测大模型效果的逻辑是同构的。

## 数据集和标注 pipeline

2. **nuScenes: A multimodal dataset for autonomous driving**（Caesar et al., 2020）  
   链接：https://arxiv.org/abs/1903.11027  
   看点：数据集论文本身就是工程细节的最佳来源——传感器配置、标注流程、质量控制。

3. **Waymo Open Dataset**（Sun et al., 2020）  
   链接：https://arxiv.org/abs/1912.04838  
   看点：和 nuScenes 对比读，理解不同团队在采集标准上的设计取舍。

## 减少对人工标注的依赖

4. **UniAD: Planning-oriented Autonomous Driving**（Hu et al., 2023，NeurIPS Best Paper）  
   链接：https://arxiv.org/abs/2212.10156  
   看点：端到端多任务模型，数据处理上的启示是：多任务联合标注比分别标注更高效，一份数据同时监督感知/预测/规划。

5. **OccupancyDreamer / DriveDreamer**（2023–2024）  
   链接：https://arxiv.org/abs/2309.09777（DriveDreamer）  
   看点：用生成模型合成驾驶场景数据，缩小 sim-to-real gap——和 Phi 系列合成文本数据的逻辑完全相同，是合成数据思路在自动驾驶的对应。

## 场景多样性

6. **ScenarioNet: Open-Source Platform for Large-Scale Traffic Scenario Simulation and Modeling**（2023）  
   链接：https://arxiv.org/abs/2306.12241  
   看点：统一多个数据集格式，研究场景多样性对模型泛化的影响。

## 两个领域的核心共同点

| | LLM | 自动驾驶 |
|--|-----|---------|
| 质量过滤 | 困惑度 + fastText | 模型不确定性 + 人工复核 |
| 数据飞轮 | 更好模型 → 更好过滤 | 部署 → 发现 corner case → 标注 |
| 合成数据 | LLM 生成文本（Phi） | 生成模型合成场景（DriveDreamer） |
| 少量高质量 | LIMO/LIMA | Tesla 数据引擎精选 hard case |

先把 LLM 这条线（DoReMi + Phi-1/3 + LIMA）读完，再看自动驾驶，会发现是同一套思维的不同模态应用。

### 展开：自动驾驶的质量过滤——模型不确定性 + 人工复核

LLM 用困惑度（KenLM 打分）过滤低质量文本；自动驾驶没有"困惑度"这样的通用信号，但有一个等价物：**模型自己不确定的地方，就是需要人工关注的地方**。

具体做法：
- **感知不确定性**：目标检测模型对某帧的预测置信度低于阈值（比如 3D box 置信度 < 0.5），或者多个模型版本对同一帧的预测不一致——这帧被标记为"模型不确定"。
- **预测不确定性**：轨迹预测模型对某辆车给出多个高概率但互相矛盾的轨迹——说明这个场景超出了训练分布。
- **规划异常**：规划模块触发了 fallback（降速、急刹），或者人类接管了车辆控制——这是最强的"模型失败"信号。

这些信号自动触发人工复核队列：标注员只看被标记的帧，不需要审查所有数据。**效率比例大约是 1:100**——1 个标注员处理的有效数据量，相当于随机抽查时的 100 倍。

这就是"模型不确定性 + 人工复核"的完整含义：模型作为过滤器，人工作为质量保证的最后一关。

### 展开：自动驾驶数据飞轮——部署 → 发现 corner case → 标注

这是自动驾驶数据工程最核心的机制，Tesla 在 AI Day 2021 上描述得最清楚。

飞轮的每一圈：

1. **部署**：把当前模型推送到路上的车队（Tesla 有百万量级车辆，国内头部公司也有数千到数万辆测试车）。
2. **自动挖掘**：车辆在行驶中持续上传传感器数据，后台系统自动检测"触发器"——模型不确定的帧、人类接管事件、规划异常、特定场景标签（如"夜间雨天左转"）。这一步完全自动化，每天可以从数百万公里行驶数据中筛出数千个高价值 clip。
3. **标注**：被挖掘出的 clip 进入标注队列，人工精标（3D box、轨迹、语义分割）。标注的是模型最弱的那些场景，而不是随机样本。
4. **训练**：用新标注数据 fine-tune 或重训模型，模型在这些 corner case 上变强。
5. **回到第 1 步**：更强的模型上车，能发现更细微的异常，触发下一轮更高质量的数据挖掘。

**和 LLM 数据飞轮的对比**：LLM 的飞轮是"更好的模型过滤出更好的训练数据"（DCLM 的核心逻辑），自动驾驶的飞轮是"部署到真实世界才能发现真实的失败模式"。前者是闭环的，后者必须有真实部署才能转动——这是自动驾驶数据工程比 LLM 更依赖硬件规模的根本原因。

### 展开：Tesla 数据引擎精选 hard case

Tesla 数据引擎（Data Engine）是上述飞轮的工程实现，Karpathy 在 CVPR 2021 和 Tesla AI Day 2021 中描述了核心机制：

**触发器系统（Trigger System）**：工程师定义"我想找什么场景"，系统在全车队数据里自动搜索。比如：
- "找所有在有遮挡情况下切入本车道的车辆"
- "找所有 FSD 预测和人类实际操作不一致的路口场景"
- "找所有夜间降雨中的行人横穿"

触发器可以基于感知输出、规划输出、人类接管信号，也可以基于地图信息（特定路口类型）。每个触发器定义一类 hard case。

**精选逻辑**：同一类触发器可能匹配几十万条 clip，但标注资源有限。系统会进一步按"多样性"和"难度"排序——优先标注模型最不确定、且和已有标注数据最不相似的 clip。这和 LIMA 的逻辑完全一致：**不是随机采样，而是主动找模型最弱的地方**。

**规模**：Tesla 公开过的数字是每天处理约 160 万个视频 clip，其中经过触发器筛选后进入精标的约 1 万条（筛选比例约 0.6%）。这 1 万条的标注价值，远高于随机抽取的 100 万条。

这就是"少量高质量"在自动驾驶侧的具体含义——不是数据少，而是**标注资源集中在模型最需要的地方**。
